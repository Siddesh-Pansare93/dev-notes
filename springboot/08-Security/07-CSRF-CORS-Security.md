# CSRF & CORS Security

> [!info] For the Express/TS dev
> Spring Security has CSRF protection **on by default** for state-changing methods. In a stateless JWT API, you disable it. In a cookie-session app, you keep it. Knowing the difference (and why) is the difference between a working app and a security incident.

## CSRF — what & when

**CSRF (Cross-Site Request Forgery)** exploits the browser's automatic credential sending: if your auth is cookie-based, any logged-in user visiting `evil.com` can have their browser submit a hidden form to `your-app.com` and the request looks legitimate.

The defense: a token only your app's pages know — included in form submissions, validated server-side. Spring Security automates this.

### When you NEED CSRF

- Browser-based apps using **session cookies** for auth
- Server-rendered pages with `<form>` POSTs
- Any endpoint where credentials are sent automatically by the browser

### When you can DISABLE CSRF

- **Stateless API** (no cookies; auth via `Authorization: Bearer …`)
- Mobile/native clients only
- Endpoint not callable from a browser

The reasoning: CSRF only works when the browser auto-attaches credentials. With JWT in `Authorization` header, JS code must explicitly add it — same-origin policy + CORS prevent cross-origin JS from doing so.

## Code example

### Default behavior

```java
@Bean
public SecurityFilterChain chain(HttpSecurity http) throws Exception {
    return http
        .authorizeHttpRequests(a -> a.anyRequest().authenticated())
        .formLogin(Customizer.withDefaults())
        .build();    // CSRF on by default
}
```

Forms generated by Spring's `_csrf` model attribute will include the token automatically. SPA clients must read the token from a cookie (`XSRF-TOKEN`) and echo it back as a header (`X-XSRF-TOKEN`).

### CSRF for SPA + cookies

```java
@Bean
public SecurityFilterChain chain(HttpSecurity http) throws Exception {
    return http
        .csrf(csrf -> csrf
            .csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse())
            .csrfTokenRequestHandler(new SpaCsrfTokenRequestHandler()))
        .authorizeHttpRequests(a -> a.anyRequest().authenticated())
        .build();
}
```

Spring sets `XSRF-TOKEN` cookie (readable by JS), expects `X-XSRF-TOKEN` header back. Axios does this automatically by default.

### Disable CSRF (stateless API)

```java
@Bean
public SecurityFilterChain chain(HttpSecurity http) throws Exception {
    return http
        .csrf(csrf -> csrf.disable())
        .sessionManagement(s -> s.sessionCreationPolicy(STATELESS))
        .oauth2ResourceServer(o -> o.jwt(Customizer.withDefaults()))
        .authorizeHttpRequests(a -> a.anyRequest().authenticated())
        .build();
}
```

### Selectively disable CSRF

```java
.csrf(csrf -> csrf
    .ignoringRequestMatchers("/api/webhooks/**", "/api/public/**"))
```

Useful when most of the app is form-based but specific endpoints are called server-to-server.

## Security headers

Spring sets sensible defaults. Customize:

```java
http.headers(headers -> headers
    .frameOptions(f -> f.deny())                                  // X-Frame-Options
    .contentTypeOptions(Customizer.withDefaults())                // X-Content-Type-Options: nosniff
    .httpStrictTransportSecurity(hsts -> hsts
        .maxAgeInSeconds(63072000)
        .includeSubDomains(true)
        .preload(true))
    .referrerPolicy(r -> r.policy(ReferrerPolicy.STRICT_ORIGIN_WHEN_CROSS_ORIGIN))
    .permissionsPolicy(p -> p.policy("geolocation=(), microphone=()"))
    .contentSecurityPolicy(csp -> csp
        .policyDirectives("default-src 'self'; script-src 'self' 'sha256-...'; "
                        + "style-src 'self' 'unsafe-inline'; "
                        + "frame-ancestors 'none'; "
                        + "object-src 'none'")));
```

| Header | Purpose |
| --- | --- |
| `X-Frame-Options: DENY` | Prevent clickjacking |
| `X-Content-Type-Options: nosniff` | Prevent MIME sniffing |
| `Strict-Transport-Security` | Force HTTPS for the entire domain |
| `Referrer-Policy` | Limit Referer leakage |
| `Permissions-Policy` | Disable browser features (camera, geo, etc.) |
| `Content-Security-Policy` | Whitelist sources for scripts/styles/etc. |

## CORS recap

See [[08-CORS]] for the full story. Summary for security context:

- CORS is browser-enforced. CLI tools ignore it.
- It is NOT a substitute for auth/CSRF. A misconfigured CORS doesn't prevent CSRF; it just allows the browser to read the response.
- `Access-Control-Allow-Origin: *` + credentials is invalid (and dangerous).

## Express/TS comparison

```ts
// Express
import csurf from 'csurf';
import helmet from 'helmet';

app.use(helmet());                 // security headers
app.use(cookieParser());
app.use(csurf({ cookie: true }));  // CSRF protection

app.get('/csrf', (req, res) => res.json({ token: req.csrfToken() }));
```

| Express | Spring Security |
| --- | --- |
| `csurf` middleware | `.csrf(...)` (on by default) |
| `helmet()` | `.headers(...)` (sensible defaults) |
| Disable CSRF on `/api` | `.csrf(c -> c.ignoringRequestMatchers("/api/**"))` |
| `cors()` | `.cors(...)` + `CorsConfigurationSource` ([[08-CORS]]) |
| `helmet.contentSecurityPolicy({ directives })` | `.contentSecurityPolicy(...)` |

## Decision flowchart

```
Is auth via cookies/session?
├── YES → CSRF ON. Configure CookieCsrfTokenRepository.
└── NO (Authorization header / JWT)
    └── Is the endpoint callable cross-site by a browser?
        ├── NO → Safe to disable CSRF.
        └── YES → Reconsider. Why is a token in a cookie?
```

## Gotchas

> [!danger] Disabling CSRF "to make Postman work"
> If your API is for browsers, this introduces real risk. Postman doesn't care about CSRF; the token is for browsers. Make Postman attach the `X-XSRF-TOKEN` header instead.

> [!warning] Mixed cookie + header auth
> If you accept BOTH `Cookie: SESSION=...` AND `Authorization: Bearer ...` simultaneously, you're vulnerable. Pick one auth flow per endpoint.

> [!warning] CSRF tokens with non-idempotent GETs
> CSRF only protects state-changing methods (POST/PUT/PATCH/DELETE) by default. If your `GET /transfer?to=...&amount=...` mutates state, fix the API — don't expand CSRF to GETs.

> [!warning] HSTS preload is a one-way door
> `preload` directive submits your domain to the browser HSTS list. Removing it later takes weeks/months. Test thoroughly first.

> [!warning] CSP `'unsafe-inline'` defeats most of the protection
> Take the time to use nonces or hashes for inline scripts. Many security audits flag `unsafe-inline` as critical.

> [!tip] CORS allows the browser to READ the response; CSRF prevents the request being sent
> Different threats. You generally need BOTH configured correctly for browser apps.

> [!tip] Test with `OWASP ZAP` or `Burp`
> Free tools that hit your endpoints and report missing headers / CSRF gaps.
