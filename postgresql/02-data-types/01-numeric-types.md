# Numeric Types

## Theory

PostgreSQL provides a rich set of numeric data types to handle different types of numerical data efficiently. Understanding these types is crucial for optimal database design, storage efficiency, and preventing data overflow issues.

### Numeric Type Categories

1. **Integer Types**: Whole numbers without decimal points
   - SMALLINT: 2 bytes, range -32,768 to 32,767
   - INTEGER (INT): 4 bytes, range -2,147,483,648 to 2,147,483,647
   - BIGINT: 8 bytes, range -9,223,372,036,854,775,808 to 9,223,372,036,854,775,807

2. **Arbitrary Precision**: Exact decimal numbers
   - NUMERIC(precision, scale): User-specified precision
   - DECIMAL(precision, scale): Alias for NUMERIC

3. **Floating-Point Types**: Approximate decimal numbers
   - REAL: 4 bytes, 6 decimal digits precision
   - DOUBLE PRECISION: 8 bytes, 15 decimal digits precision

4. **Auto-Incrementing Types**: Serial types for primary keys
   - SMALLSERIAL: Auto-incrementing SMALLINT (2 bytes)
   - SERIAL: Auto-incrementing INTEGER (4 bytes)
   - BIGSERIAL: Auto-incrementing BIGINT (8 bytes)
   - GENERATED ALWAYS AS IDENTITY: SQL standard alternative to SERIAL

### Storage Considerations

- Choose the smallest type that fits your data range
- NUMERIC has variable storage (overhead for precision)
- Floating-point types are faster but less precise
- Integer arithmetic is fastest

### When to Use What

- **SMALLINT**: Age, small counts, status codes
- **INTEGER**: Most general-purpose counters, IDs
- **BIGINT**: Large datasets, timestamps in milliseconds
- **NUMERIC**: Money, scientific calculations requiring exactness
- **REAL/DOUBLE PRECISION**: Scientific data, coordinates (when approximation is acceptable)
- **SERIAL/IDENTITY**: Primary keys, auto-incrementing IDs

## Syntax

### Integer Types

```sql
-- Basic integer column definitions
CREATE TABLE table_name (
    small_col SMALLINT,
    int_col INTEGER,          -- or INT
    big_col BIGINT
);
```

### Arbitrary Precision Types

```sql
-- NUMERIC/DECIMAL with precision and scale
CREATE TABLE products (
    price NUMERIC(10, 2),     -- 10 total digits, 2 after decimal
    weight DECIMAL(8, 3),     -- 8 total digits, 3 after decimal
    exact_value NUMERIC       -- No limit (slow, avoid if possible)
);
```

### Floating-Point Types

```sql
CREATE TABLE measurements (
    temperature REAL,
    distance DOUBLE PRECISION
);
```

### Serial Types (Legacy)

```sql
CREATE TABLE orders (
    order_id SERIAL PRIMARY KEY,
    customer_id INTEGER,
    total NUMERIC(10, 2)
);
```

### GENERATED ALWAYS AS IDENTITY (Recommended)

```sql
CREATE TABLE customers (
    customer_id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name VARCHAR(100)
);

-- With options
CREATE TABLE products (
    product_id INTEGER GENERATED ALWAYS AS IDENTITY (
        START WITH 1000
        INCREMENT BY 1
        MINVALUE 1000
        MAXVALUE 999999
        CACHE 20
    ) PRIMARY KEY,
    product_name VARCHAR(200)
);

-- GENERATED BY DEFAULT (allows manual insertion)
CREATE TABLE logs (
    log_id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    message TEXT
);
```

## Examples

### Basic Integer Usage

```sql
-- Create a table with different integer types
CREATE TABLE inventory (
    item_id INTEGER PRIMARY KEY,
    warehouse_section SMALLINT,
    quantity INTEGER,
    global_stock BIGINT
);

-- Insert data
INSERT INTO inventory (item_id, warehouse_section, quantity, global_stock)
VALUES
    (1, 5, 150, 1500000),
    (2, 12, 89, 890000),
    (3, 3, 0, 0);

-- Query
SELECT * FROM inventory WHERE quantity > 0;

-- Arithmetic operations
SELECT
    item_id,
    quantity,
    quantity * 2 AS doubled,
    quantity + 10 AS increased
FROM inventory;
```

### NUMERIC/DECIMAL for Precision

```sql
-- Financial data requiring exact precision
CREATE TABLE transactions (
    transaction_id SERIAL PRIMARY KEY,
    amount NUMERIC(12, 2),
    tax_rate NUMERIC(5, 4),
    calculated_tax NUMERIC(12, 2)
);

INSERT INTO transactions (amount, tax_rate)
VALUES
    (100.00, 0.0875),
    (1599.99, 0.0625),
    (49.50, 0.1000);

-- Calculate tax (exact precision)
UPDATE transactions
SET calculated_tax = amount * tax_rate;

SELECT
    amount,
    tax_rate,
    calculated_tax,
    amount + calculated_tax AS total
FROM transactions;

-- Rounding
SELECT
    amount,
    ROUND(amount * tax_rate, 2) AS rounded_tax,
    TRUNC(amount * tax_rate, 2) AS truncated_tax,
    CEIL(amount * tax_rate) AS ceiling_tax,
    FLOOR(amount * tax_rate) AS floor_tax
FROM transactions;
```

### Floating-Point Types

```sql
-- Scientific measurements (approximation acceptable)
CREATE TABLE sensor_readings (
    reading_id SERIAL PRIMARY KEY,
    sensor_name VARCHAR(50),
    temperature REAL,
    pressure DOUBLE PRECISION,
    timestamp TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP
);

INSERT INTO sensor_readings (sensor_name, temperature, pressure)
VALUES
    ('Sensor-A', 23.456, 1013.25678901234),
    ('Sensor-B', -5.123, 980.12345678901),
    ('Sensor-C', 100.789, 1020.98765432109);

-- Floating-point arithmetic
SELECT
    sensor_name,
    temperature,
    temperature * 9/5 + 32 AS fahrenheit,
    pressure,
    CAST(pressure AS REAL) AS pressure_real
FROM sensor_readings;

-- Demonstrating floating-point imprecision
SELECT
    0.1::REAL + 0.2::REAL AS real_sum,
    0.1::DOUBLE PRECISION + 0.2::DOUBLE PRECISION AS double_sum,
    0.1::NUMERIC + 0.2::NUMERIC AS numeric_sum;
```

### SERIAL vs GENERATED ALWAYS AS IDENTITY

```sql
-- Traditional SERIAL approach
CREATE TABLE old_style_users (
    user_id SERIAL PRIMARY KEY,
    username VARCHAR(50)
);

-- Modern IDENTITY approach (SQL standard)
CREATE TABLE new_style_users (
    user_id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    username VARCHAR(50)
);

-- Insert data (both work the same for basic inserts)
INSERT INTO old_style_users (username) VALUES ('alice'), ('bob');
INSERT INTO new_style_users (username) VALUES ('charlie'), ('david');

-- View sequences
SELECT * FROM old_style_users;
SELECT * FROM new_style_users;

-- GENERATED ALWAYS prevents manual ID insertion (safer)
-- This will fail:
-- INSERT INTO new_style_users (user_id, username) VALUES (999, 'eve');

-- GENERATED BY DEFAULT allows manual insertion
CREATE TABLE flexible_users (
    user_id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    username VARCHAR(50)
);

INSERT INTO flexible_users (username) VALUES ('user1');
INSERT INTO flexible_users (user_id, username) VALUES (100, 'user100');
INSERT INTO flexible_users (username) VALUES ('user101');

SELECT * FROM flexible_users;
```

### Overflow Behavior

```sql
-- Demonstrating overflow protection
CREATE TABLE overflow_test (
    small_val SMALLINT,
    int_val INTEGER,
    big_val BIGINT
);

-- This will work
INSERT INTO overflow_test (small_val) VALUES (32767);

-- This will fail with overflow error
-- INSERT INTO overflow_test (small_val) VALUES (32768);

-- Safe casting with error handling
DO $$
BEGIN
    INSERT INTO overflow_test (small_val) VALUES (32768);
EXCEPTION WHEN numeric_value_out_of_range THEN
    RAISE NOTICE 'Value too large for SMALLINT';
END $$;

-- Checking ranges
SELECT
    pg_typeof(32767::SMALLINT) AS type,
    32767::SMALLINT AS max_smallint;
```

### Precision and Scale Examples

```sql
-- Understanding precision and scale
CREATE TABLE precision_examples (
    id SERIAL PRIMARY KEY,
    value NUMERIC(5, 2),  -- 5 total digits, 2 after decimal
    description TEXT
);

-- Valid insertions
INSERT INTO precision_examples (value, description) VALUES
    (123.45, 'Maximum valid value'),
    (999.99, 'Near maximum'),
    (0.01, 'Minimum positive'),
    (-999.99, 'Negative maximum');

-- These will fail (exceeds precision)
-- INSERT INTO precision_examples (value) VALUES (1000.00);  -- Too many digits
-- INSERT INTO precision_examples (value) VALUES (123.456); -- Too many decimal places

-- Automatic rounding
CREATE TABLE auto_round (
    value NUMERIC(5, 2)
);

INSERT INTO auto_round VALUES (123.456);  -- Rounds to 123.46
SELECT * FROM auto_round;

-- Scale zero for whole numbers
CREATE TABLE whole_numbers (
    amount NUMERIC(10, 0)  -- No decimal places
);

INSERT INTO whole_numbers VALUES (1234567890);
```

### Practical Application: E-commerce

```sql
-- Complete e-commerce example
CREATE TABLE products (
    product_id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    sku VARCHAR(20) UNIQUE,
    name VARCHAR(200),
    price NUMERIC(10, 2),
    cost NUMERIC(10, 2),
    weight_kg NUMERIC(8, 3),
    stock_quantity INTEGER,
    total_sold BIGINT DEFAULT 0
);

CREATE TABLE order_items (
    order_item_id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    product_id INTEGER REFERENCES products(product_id),
    quantity SMALLINT,
    unit_price NUMERIC(10, 2),
    discount_percent NUMERIC(5, 2) DEFAULT 0.00,
    line_total NUMERIC(12, 2)
);

-- Insert products
INSERT INTO products (sku, name, price, cost, weight_kg, stock_quantity)
VALUES
    ('LAPTOP-001', 'Premium Laptop', 1299.99, 899.99, 2.500, 50),
    ('MOUSE-042', 'Wireless Mouse', 29.99, 12.50, 0.150, 500),
    ('DESK-LAMP', 'LED Desk Lamp', 49.99, 22.00, 1.200, 100);

-- Create order
INSERT INTO order_items (product_id, quantity, unit_price, discount_percent)
VALUES
    (1, 2, 1299.99, 10.00),
    (2, 5, 29.99, 0.00),
    (3, 1, 49.99, 5.00);

-- Calculate line totals
UPDATE order_items
SET line_total = quantity * unit_price * (1 - discount_percent / 100);

-- Query order summary
SELECT
    oi.order_item_id,
    p.name,
    oi.quantity,
    oi.unit_price,
    oi.discount_percent,
    oi.line_total,
    (oi.line_total - (oi.quantity * p.cost)) AS profit
FROM order_items oi
JOIN products p ON oi.product_id = p.product_id;

-- Aggregate calculations
SELECT
    SUM(line_total) AS order_total,
    AVG(line_total) AS avg_item_total,
    COUNT(*) AS item_count
FROM order_items;
```

## Common Mistakes

### 1. Using Wrong Integer Type

```sql
-- MISTAKE: Using BIGINT unnecessarily
CREATE TABLE users (
    user_id BIGINT PRIMARY KEY  -- Overkill for most applications
);

-- BETTER: Use INTEGER unless you need > 2 billion records
CREATE TABLE users_better (
    user_id INTEGER PRIMARY KEY
);
```

### 2. Using Floating-Point for Money

```sql
-- MISTAKE: Using REAL/DOUBLE for currency
CREATE TABLE bad_transactions (
    amount REAL  -- Imprecise!
);

INSERT INTO bad_transactions VALUES (0.1 + 0.2);
SELECT * FROM bad_transactions;  -- May not equal 0.3 exactly

-- CORRECT: Use NUMERIC for money
CREATE TABLE good_transactions (
    amount NUMERIC(10, 2)
);
```

### 3. Forgetting Scale in NUMERIC

```sql
-- MISTAKE: Not specifying precision and scale
CREATE TABLE products_bad (
    price NUMERIC  -- Variable, slower, unpredictable storage
);

-- BETTER: Always specify precision and scale
CREATE TABLE products_good (
    price NUMERIC(10, 2)
);
```

### 4. Incorrect Precision/Scale

```sql
-- MISTAKE: Scale larger than precision
-- CREATE TABLE wrong (
--     value NUMERIC(2, 5)  -- Invalid!
-- );

-- CORRECT: Scale must be <= precision
CREATE TABLE correct (
    value NUMERIC(8, 5)  -- 8 total digits, 5 after decimal
);
```

### 5. Not Handling Overflow

```sql
-- MISTAKE: Assuming unlimited range
CREATE TABLE counters (
    visit_count INTEGER
);

-- This will eventually overflow if incremented 2+ billion times

-- BETTER: Choose appropriate type or add overflow protection
CREATE TABLE counters_safe (
    visit_count BIGINT
);
```

### 6. Using SERIAL Incorrectly

```sql
-- MISTAKE: Trying to control SERIAL values manually
CREATE TABLE items (
    id SERIAL PRIMARY KEY
);

-- This works but can cause sequence drift
INSERT INTO items (id) VALUES (999);
INSERT INTO items DEFAULT VALUES;  -- Might conflict later

-- BETTER: Use GENERATED ALWAYS AS IDENTITY
CREATE TABLE items_better (
    id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY
);

-- This prevents manual insertion, keeping sequence consistent
```

## Best Practices

### 1. Choose the Right Type

```sql
-- Use smallest type that fits your data
CREATE TABLE user_preferences (
    user_id INTEGER,
    theme_id SMALLINT,           -- Only a few themes
    font_size SMALLINT,          -- 8-72 range
    notifications_enabled BOOLEAN -- See 04-boolean-uuid-enum.md
);
```

### 2. Always Specify Precision for NUMERIC

```sql
-- Good practice
CREATE TABLE financial_data (
    amount NUMERIC(12, 2),       -- Explicit precision
    tax_rate NUMERIC(5, 4),      -- e.g., 0.0875
    quantity NUMERIC(10, 0)      -- Whole numbers only
);
```

### 3. Use IDENTITY Over SERIAL

```sql
-- Modern approach (SQL standard, better control)
CREATE TABLE entities (
    id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY
);

-- Use BY DEFAULT if you need occasional manual insertion
CREATE TABLE imported_data (
    id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY
);
```

### 4. Add Constraints for Data Validation

```sql
CREATE TABLE products (
    product_id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    price NUMERIC(10, 2) CHECK (price >= 0),
    quantity INTEGER CHECK (quantity >= 0),
    discount_percent NUMERIC(5, 2) CHECK (discount_percent BETWEEN 0 AND 100)
);
```

### 5. Document Your Type Choices

```sql
CREATE TABLE sensor_data (
    sensor_id SMALLINT,          -- Max 100 sensors in facility
    reading DOUBLE PRECISION,     -- Scientific precision acceptable
    precise_calibration NUMERIC(10, 6)  -- Calibration needs exactness
);

COMMENT ON COLUMN sensor_data.reading IS 'Raw sensor reading, approximate';
COMMENT ON COLUMN sensor_data.precise_calibration IS 'Exact calibration value';
```

### 6. Use Type Casting Explicitly

```sql
-- Be explicit about conversions
SELECT
    5::NUMERIC / 2::NUMERIC AS numeric_division,  -- 2.5
    5::INTEGER / 2::INTEGER AS integer_division,  -- 2
    5::REAL / 2::REAL AS float_division;          -- 2.5
```

## Practice Exercises

### Exercise 1: Product Inventory System

Create a comprehensive product inventory system that tracks:
- Products with auto-incrementing IDs
- Prices with exact precision
- Stock quantities
- Warehouse locations (limited number)
- Total units sold (potentially very large)

Requirements:
1. Create appropriate tables with optimal numeric types
2. Insert sample data for at least 5 products
3. Write a query that calculates total inventory value
4. Create a query showing products with low stock (quantity < 10)
5. Calculate profit margin percentage for each product

<details>
<summary>Solution</summary>

```sql
-- Create tables
CREATE TABLE products_inventory (
    product_id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    product_name VARCHAR(200) NOT NULL,
    sku VARCHAR(50) UNIQUE NOT NULL,
    cost_price NUMERIC(10, 2) CHECK (cost_price >= 0),
    selling_price NUMERIC(10, 2) CHECK (selling_price >= cost_price),
    stock_quantity INTEGER CHECK (stock_quantity >= 0),
    warehouse_section SMALLINT CHECK (warehouse_section BETWEEN 1 AND 50),
    total_units_sold BIGINT DEFAULT 0
);

-- Insert sample data
INSERT INTO products_inventory
    (product_name, sku, cost_price, selling_price, stock_quantity, warehouse_section, total_units_sold)
VALUES
    ('Laptop Pro 15"', 'LPT-PRO-15', 899.99, 1299.99, 25, 5, 1250000),
    ('Wireless Mouse', 'MOUSE-WL-01', 12.50, 29.99, 150, 12, 5000000),
    ('USB-C Cable 2m', 'CABLE-UC-2M', 3.00, 9.99, 8, 12, 10000000),
    ('Monitor 27"', 'MON-27-4K', 299.99, 499.99, 50, 5, 750000),
    ('Keyboard RGB', 'KBD-RGB-MEC', 45.00, 89.99, 75, 8, 2500000);

-- Total inventory value
SELECT
    SUM(stock_quantity * cost_price) AS total_cost_value,
    SUM(stock_quantity * selling_price) AS total_retail_value,
    SUM(stock_quantity * (selling_price - cost_price)) AS potential_profit
FROM products_inventory;

-- Low stock products
SELECT
    product_name,
    sku,
    stock_quantity,
    warehouse_section
FROM products_inventory
WHERE stock_quantity < 10
ORDER BY stock_quantity;

-- Profit margin calculation
SELECT
    product_name,
    cost_price,
    selling_price,
    selling_price - cost_price AS profit_per_unit,
    ROUND(((selling_price - cost_price) / selling_price * 100), 2) AS profit_margin_percent,
    total_units_sold
FROM products_inventory
ORDER BY profit_margin_percent DESC;
```

### Exercise 2: Financial Transactions

Create a banking transaction system that handles:
- Account balances with exact precision
- Transaction amounts
- Interest rates with 4 decimal places
- Transaction counts

Requirements:
1. Create accounts and transactions tables
2. Insert sample accounts with initial balances
3. Record 5-10 transactions (deposits and withdrawals)
4. Calculate total deposits, withdrawals, and current balance per account
5. Apply interest to all accounts (use a stored procedure or update statement)

<details>
<summary>Solution</summary>

```sql
-- Create tables
CREATE TABLE bank_accounts (
    account_id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    account_number VARCHAR(20) UNIQUE NOT NULL,
    account_holder VARCHAR(100) NOT NULL,
    balance NUMERIC(15, 2) DEFAULT 0.00 CHECK (balance >= 0),
    interest_rate NUMERIC(5, 4) DEFAULT 0.0250,
    transaction_count BIGINT DEFAULT 0
);

CREATE TABLE transactions (
    transaction_id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    account_id INTEGER REFERENCES bank_accounts(account_id),
    transaction_type VARCHAR(10) CHECK (transaction_type IN ('DEPOSIT', 'WITHDRAWAL')),
    amount NUMERIC(15, 2) CHECK (amount > 0),
    transaction_date TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP
);

-- Insert accounts
INSERT INTO bank_accounts (account_number, account_holder, balance, interest_rate)
VALUES
    ('ACC-001', 'Alice Johnson', 5000.00, 0.0325),
    ('ACC-002', 'Bob Smith', 15000.00, 0.0275),
    ('ACC-003', 'Carol White', 2500.00, 0.0250);

-- Record transactions
INSERT INTO transactions (account_id, transaction_type, amount)
VALUES
    (1, 'DEPOSIT', 1000.00),
    (1, 'WITHDRAWAL', 500.00),
    (2, 'DEPOSIT', 5000.00),
    (2, 'WITHDRAWAL', 2000.00),
    (3, 'DEPOSIT', 750.00),
    (3, 'DEPOSIT', 250.00),
    (1, 'WITHDRAWAL', 200.00);

-- Calculate summary per account
SELECT
    ba.account_number,
    ba.account_holder,
    ba.balance AS current_balance,
    COALESCE(SUM(CASE WHEN t.transaction_type = 'DEPOSIT' THEN t.amount END), 0) AS total_deposits,
    COALESCE(SUM(CASE WHEN t.transaction_type = 'WITHDRAWAL' THEN t.amount END), 0) AS total_withdrawals,
    COUNT(t.transaction_id) AS transaction_count
FROM bank_accounts ba
LEFT JOIN transactions t ON ba.account_id = t.account_id
GROUP BY ba.account_id, ba.account_number, ba.account_holder, ba.balance;

-- Apply interest (annual interest calculated)
UPDATE bank_accounts
SET balance = balance + (balance * interest_rate);

SELECT
    account_number,
    account_holder,
    balance AS new_balance,
    interest_rate
FROM bank_accounts;
```

### Exercise 3: Type Selection Challenge

Given the following scenarios, choose the most appropriate numeric type and justify your choice:

1. A counter for website page views that starts at 0
2. The price of a product in an e-commerce store
3. Coordinates for GPS location (latitude/longitude)
4. Age of users in years
5. Scientific measurement requiring 12 decimal places
6. Primary key for a table expected to have 50 million rows
7. Percentage values (0-100 with 2 decimal places)
8. File size in bytes

<details>
<summary>Solution</summary>

```sql
-- 1. Website page views counter
CREATE TABLE page_views (
    page_id INTEGER,
    view_count BIGINT DEFAULT 0  -- Can grow very large over time
);
-- Justification: BIGINT handles billions of views safely

-- 2. Product price
CREATE TABLE products (
    product_id INTEGER,
    price NUMERIC(10, 2)  -- Exact precision for money
);
-- Justification: NUMERIC for exact decimal representation, no rounding errors

-- 3. GPS coordinates
CREATE TABLE locations (
    location_id INTEGER,
    latitude DOUBLE PRECISION,   -- -90 to 90
    longitude DOUBLE PRECISION   -- -180 to 180
);
-- Justification: Floating-point acceptable, range fits, precision adequate

-- 4. User age
CREATE TABLE users (
    user_id INTEGER,
    age SMALLINT CHECK (age BETWEEN 0 AND 150)
);
-- Justification: SMALLINT sufficient, range -32,768 to 32,767

-- 5. Scientific measurement
CREATE TABLE measurements (
    measurement_id INTEGER,
    value NUMERIC(20, 12)  -- High precision required
);
-- Justification: NUMERIC for exact precision, 12 decimal places

-- 6. Primary key for 50M rows
CREATE TABLE large_table (
    id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY  -- Handles 2.1B
);
-- Justification: INTEGER sufficient for 50M (max 2.1B), BIGINT if future growth expected

-- 7. Percentage values
CREATE TABLE statistics (
    stat_id INTEGER,
    completion_percent NUMERIC(5, 2) CHECK (completion_percent BETWEEN 0 AND 100)
);
-- Justification: NUMERIC(5, 2) allows 0.00 to 100.00 exactly

-- 8. File size in bytes
CREATE TABLE files (
    file_id INTEGER,
    size_bytes BIGINT  -- Files can exceed 4GB
);
-- Justification: BIGINT for files > 2GB (INTEGER max ~2GB)
```

</details>
