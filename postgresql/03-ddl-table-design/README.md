# DDL & Table Design

Data Definition Language (DDL) is the backbone of any PostgreSQL database ŌĆö it determines how your data is structured, what values are allowed, and how tables relate to one another. This section gives you the skills to design tables that are correct by construction, from basic `CREATE TABLE` through constraints, computed columns, inheritance, and specialised table types.

## Learning Path

### Beginner
Start here if you are new to PostgreSQL schema design or coming from another relational database:

1. **Chapter 1 ŌĆö CREATE, ALTER, DROP** ŌĆö Learn the fundamental DDL commands, column data types, and how to safely evolve a schema over time with `ALTER TABLE`.
2. **Chapter 2 ŌĆö Constraints** ŌĆö Understand `PRIMARY KEY`, `FOREIGN KEY`, `UNIQUE`, `CHECK`, and `NOT NULL`. Learn how ON DELETE actions (CASCADE, RESTRICT, SET NULL) protect referential integrity.
3. **Chapter 3 ŌĆö Default Values and Generated Columns** ŌĆö Reduce boilerplate in your application by letting the database supply timestamps, UUIDs, auto-increments, and computed fields automatically.

### Intermediate
Continue once you are comfortable writing schema migrations and working with multi-table designs:

4. **Chapter 4 ŌĆö Table Inheritance** ŌĆö Explore PostgreSQL's native table inheritance to share column definitions across related tables and model hierarchical data structures.
5. **Chapter 5 ŌĆö Temporary and Unlogged Tables** ŌĆö Learn when to reach for session-scoped temporary tables or high-throughput unlogged tables and understand the trade-offs involved.

### Advanced
Revisit the earlier chapters with a focus on production-quality patterns:

- Re-read **Chapter 2** focusing on named constraints, `EXCLUDE` constraints with GiST indexes, and `DEFERRABLE` foreign keys for complex bulk-load scenarios.
- Re-read **Chapter 3** with attention to `GENERATED ALWAYS AS IDENTITY` vs `SERIAL`, sequence management, and UUID defaults for distributed systems.
- Study **Chapter 4** to understand query planning implications and how partitioning (covered in a later module) supersedes inheritance for most large-table use cases.

## What You'll Learn

- Write idempotent `CREATE TABLE IF NOT EXISTS` and `DROP TABLE IF EXISTS` scripts suitable for repeatable deployments
- Use `ALTER TABLE` to add, drop, rename, and retype columns on live databases without data loss
- Apply all six PostgreSQL constraint types ŌĆö `PRIMARY KEY`, `FOREIGN KEY`, `UNIQUE`, `CHECK`, `NOT NULL`, and `EXCLUDE` ŌĆö and name them consistently for easy maintenance
- Choose the right ON DELETE action (`CASCADE`, `RESTRICT`, `SET NULL`, `SET DEFAULT`) for each foreign key relationship
- Defer constraint validation across transaction boundaries with `DEFERRABLE INITIALLY DEFERRED`
- Set static, expression-based, and function-based column defaults (timestamps, UUIDs, random values)
- Define stored `GENERATED ALWAYS AS` columns for computed fields such as totals, full names, and derived flags
- Distinguish between `GENERATED ALWAYS AS IDENTITY`, `GENERATED BY DEFAULT AS IDENTITY`, and `SERIAL`, and know when to use each
- Manage sequences: restart, advance, and sync after bulk data migrations
- Model hierarchical or shared-schema data using table inheritance
- Use temporary tables for session-scoped scratch space and unlogged tables for high-speed staging data
- Wrap complex schema migrations in transactions so that partial failures leave the database unchanged

## Prerequisites

Before working through this section you should be comfortable with:

No prior DDL experience is required ŌĆö the section starts from first principles.

## How to Use This Guide

1. **Follow the numbered order.** Each chapter builds on the previous one. Constraints are much easier to understand after you can write a `CREATE TABLE` statement, and generated columns make more sense once you know how defaults work.
2. **Run every example yourself.** Each file contains working SQL you can copy directly into `psql`. Seeing constraint violations in real time is the fastest way to internalise the rules.
3. **Do the practice exercises.** Every chapter ends with three progressively harder exercises. Resist the urge to skip them ŌĆö the exercises are where the concepts actually stick.
4. **Read the Common Mistakes sections carefully.** They cover the exact errors you will encounter in real projects: dropping columns with dependencies, adding NOT NULL to columns that already have NULLs, and sequences falling out of sync after manual inserts.
5. **Use the cross-references.** Each chapter links to related topics in other modules (indexes, transactions, partitioning). Follow those links when you are ready to go deeper.

Good schema design pays dividends for the entire lifetime of a project ŌĆö take the time to get it right from the start.
